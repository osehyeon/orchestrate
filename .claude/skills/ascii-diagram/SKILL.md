# ASCII Diagram Skill

모노스페이스 폰트로 렌더링되는 markdown 코드 블록 안에 박스(`+--+`)와 흐름선(`|` `v` `<--`)으로 다이어그램을 그릴 때의 규칙. 본 스킬은 실패 사례에서 추출한 13개 규칙으로 구성된다.

## 언제 쓰는가

- 트랜스포머/MLP 블록처럼 모듈과 흐름을 보여줄 때
- 데이터/요청 파이프라인 단계 흐름
- before/after 구조 비교
- mermaid가 과한 가벼운 다이어그램

## 절대 규칙 (어기면 정렬이 깨진다)

### 1. 박스 안에는 ASCII만 — 한글/그리스 문자/⊙ 같은 wide 글리프 금지

한글은 모노스페이스에서 2-column 너비, ASCII box-drawing 문자(`+ - | v`)는 1-column. 같은 줄에 섞으면 닫는 `|`가 어긋난다.

```
[NG]
+--------+
|  RMSNorm(α_attn) |   <-- α는 1-col이지만 한글 섞이면 무너짐
+--------+

[OK — 한글 설명은 박스 밖]
+--------+
| RMSNorm |   <-- alpha_attn scale
+--------+
```

박스 안에는 영문/숫자/`_ . , ( ) [ ] < > * + - / | ^ v` 만. 한글 본문 설명은 박스 외부 또는 다이어그램 코드 블록 밖에 둔다.

### 2. Unicode box-drawing(`┌─┐│└┘`) 대신 ASCII(`+ - |`)를 써라

이유: (a) 일부 markdown 뷰어/폰트에서 정렬 깨짐, (b) 프로그래밍 폰트의 ligature가 `--`나 `<--`를 합성 글리프로 바꿔 폭이 달라질 수 있음, (c) 검사기를 짜기 쉬움(char index = visual column).

```
[NG]                  [OK]
┌─────┐              +-----+
│ Box │              | Box |
└─────┘              +-----+
```

### 3. 박스 폭 = 윗변 폭 = 모든 내부 라인 폭

`+----+`이 N+2 폭이면, 그 박스의 모든 `|...|` 라인도 정확히 N+2 폭이어야 한다. ASCII만 쓰면 character count = column count이므로 문자 수만 맞추면 된다.

```
[NG — alpha_attn(10자)이 alpha_mlp(9자) 기준 박스를 1칸 넘침]
+--------------------+
|  RMSNorm(alpha_attn)|     <-- 닫는 | 가 1칸 밀림
+----------+---------+

[OK — 박스 자체를 21로 키우거나 라벨을 줄이거나, 둘 다 패딩 통일]
+---------------------+
|  RMSNorm(alpha_attn)|
+----------+----------+
```

### 4. 진입 화살표 column = 출구 T-junction column

박스 안에서 흐름선이 점프하지 않게 하려면, 위에서 들어오는 `v`의 column과 아래에서 나가는 `+`(T-junction)의 column이 같아야 한다.

```
[NG — 입구 col 4, 출구 col 11. 박스 안에서 7칸 점프]
    v
+-----------+----------+
|  Box                 |
+-----------+----------+
            |     <-- 출구가 col 11

[OK — 입구 v와 출구 + 둘 다 col 11]
           v
+----------v----------+
|  Box                |
+----------+----------+
           |
```

### 5. 박스 위 v 마커를 박스 아래 + 마커와 짝을 맞춰라

규칙 4의 시각적 표현. 들어오는 자리에 `v`, 나가는 자리에 `+`를 넣는다. 들어오기만 하고 나가지 않거나 그 반대인 경우는 한 쪽만.

```
+----v----+        +----v----+
|  Box    |        |  Func   |
+----+----+        +---------+   <-- 출구 없는 박스(예: 종착)
```

### 6. 다중 분기는 분기 column이 각 박스의 가운데 column과 일치해야 한다

3-way split에서 split 라인의 `+`들과 각 박스의 가운데 T-junction column이 정확히 일치해야 한다. **그냥 균등 간격으로 그리지 말고, 박스 위치에서 역산하라.**

박스 3개를 col 4/22/40 중심에 두기로 정했다면, split도 `+` at cols 4, 22, 40이어야 한다.

```
       |
    +--+----------------+----------------+    <-- + at cols 4, 22, 40 (split 분기)
    |                   |                |        gap = 18 (= 박스간격)
    v                   v                v
+---v----+        +---v----+        +---v----+
|  W_q   |        |  W_k   |        |  W_v   |   <-- 박스 cols 0-9, 18-27, 36-45
| (FP16) |        | (FP16) |        | (FP16) |
+---+----+        +---+----+        +---+----+   <-- 출구 T at cols 4, 22, 40
    |                 |                 |
```

박스가 10칸 폭이고 8칸 gap이면, 박스 중심 column 간격은 `10 + 8 = 18`. split의 `+` 간격도 18이어야 한다.

### 7. residual은 ring으로 그리지 말고 `(+)` 노드 + 주석으로 표현

좌측 세로줄로 residual을 감싸면 외곽 박스 안 모든 줄의 우변 `|` 정렬을 맞춰야 해서 깨지기 쉽다. 단순한 `(+) <-- residual ...` 한 줄이면 충분.

```
[NG — residual ring]
   x
   |
   +-----------+
   |           v
   |       [Module]
   |           |
   +---------->(+)
               |

[OK — sequential + annotation]
   x
   |
   v
[Module]
   |
   v
  (+) <-- residual from x
   |
   v
```

### 8. 평행 박스는 같은 모양으로 통일

W_q, W_k, W_v 같이 동등한 N개 박스는 폭/내부 정렬이 모두 같아야 한다. `+---v----+` `+----v---+` 처럼 v 위치가 다르면 시각적 리듬이 깨진다 (그리고 보통 정렬 오류의 신호).

### 9. 비수직 흐름 connector 표준 패턴

수직이 아닌 분기/합류는 다음 패턴을 사용한다:

```
좌측에서 박스로 진입:
  ----->|  Box  |
        +-------+

우측에서 박스로 진입 (어떤 흐름이 박스 우변으로 들어옴):
        +-------+
  v     |  Box  |<-----+      <-- + 의 column이 위 v 의 column과 일치
  |     |       |      |
  v     +-------+      |
                       |   (위에서 |로 내려와 +에서 좌회전)

residual add:
  attn_out
     |
     v
    (+) <--+ residual from before-attn
     |     ^
     v    (+의 col이 들어오는 흐름의 col과 일치)
```

핵심: connector의 `+`는 들어오는 `|`의 column과 정확히 일치해야 한다. 불일치하면 흐름이 끊긴 것처럼 보인다.

### 10. label-only 노드는 박스 없이 `[Name]`

함수 호출, 활성화 함수, 유틸 연산 등 박스를 그릴 필요 없는 단계는 `[FuncName]` 으로 표기한다. 들어오는 `v`는 라벨의 가운데 글자 또는 `[`에 떨어지면 OK.

```
   |
   v
[RoPE]
   |
   v
[INT4 quantize]
   |
```

`[Name]` 의 width를 의식해서 `v`가 라벨 중앙에 오도록 column을 잡는다.

### 11. 박스 옆 주석은 `<-- text` 로 우측 밖에

박스 내부에 주석을 넣지 말고, 닫는 `|` 뒤에 두 칸 띄고 `<-- ` 다음에 주석.

```
+----v----+
| RMSNorm |  <-- alpha_attn scale already absorbed
+----+----+
```

이 주석은 박스 폭에 영향을 주지 않으며, 한글 주석도 가능하다 (박스 밖이므로).

### 12. 다이어그램은 ` ``` ` code fence로 감싸고 언어 태그를 붙이지 마라

```
\`\`\`
   x
   |
   v
+----+
| Box|
+----+
\`\`\`
```

언어 태그(예: `text`, `ascii`)를 붙이면 일부 뷰어에서 syntax highlight를 시도하다 폭이 깨질 수 있다. plain code fence가 가장 안전하다.

### 13. 본문에서 영문 기술용어는 `` `inline code` `` 로 감싸 spell-checker 회피

`QuaRot의`, `outlier가`, `Hadamard로` 같은 한글+영문 혼합은 LTeX/spell-checker가 squiggle/strikethrough로 표시한다. 다음 둘 중 하나를 택한다:

- `` `QuaRot`의 ``  ← inline code 감싸기 (spell check skip)
- 영문과 한글 사이에 공백: `QuaRot 의`  ← 자연스럽지 않음

가장 깔끔한 건 inline code. 추가 보너스로 시각적 구분도 된다.

### 14. 한국어 본문에서 `~`는 아예 쓰지 마라 — 취소선이 된다

GFM 표준은 **이중** tilde `~~text~~`를 취소선으로 정의한다. 그러나 일부 markdown 렌더러(VS Code 마크다운 플러그인 일부, markdown-it 확장 등)는 비표준 확장으로 **단일** `~text~`도 취소선으로 처리한다. 한 줄 안에 `~`가 짝수 번 등장하면 그 사이가 통째로 취소선으로 바뀐다.

한국어 글에서 "R1부터 R4까지"를 `R1~R4`로 짧게 쓰는 습관이 위험하다 — 한 줄에 `R1~R4`가 두 번 나오면 두 `~` 사이 텍스트가 모두 취소선이 된다.

```
[NG — 이중 tilde]
> R1~~R4 표기를 사용하지 않으며, R1~~R4는 편의 표기다.
     ^^^^                          ^^^^
     GFM 표준 취소선 — 모든 렌더러에서 strikethrough

[NG — 단일 tilde 두 번]
> R1~R4 표기를 사용하지 않으며, R1~R4는 편의 표기다.
     ^                          ^
     일부 렌더러(특히 VS Code preview 일부)에서 strikethrough

[OK]
> R1-R4 표기를 사용하지 않으며, R1-R4는 편의 표기다.
     ^                          ^
     하이픈은 어디서나 안전
```

규칙:
- 범위/대시 의미로는 `-` (하이픈) 또는 "에서~까지" 풀어쓰기
- 진짜 취소선이 필요할 때만 `~~text~~` (의도적으로)
- 점검: 본문에서 `grep -n '~' file.md` 실행해 의도하지 않은 `~`가 없는지 확인 (이중·단일 모두)

## 표준 패턴 모음

### 단순 수직 흐름

```
   x
   |
   v
+----v----+
|  Module |
+----+----+
   |
   v
  (+) <-- residual
   |
   v
output
```

### 3-way 분기 + 합류

```
                     |
    +----------------+-----------------+
    |                |                 |
    v                v                 v
+---v----+      +---v----+        +---v----+
|  M1    |      |  M2    |        |  M3    |
+---+----+      +---+----+        +---+----+
    |               |                 |
    +-------> joiner(M1, M2, M3) <----+
                    |
                    v
                 result
```

### 한 흐름이 옆 박스로 진입 (V → KV cache 패턴)

```
W_q col 4         W_k col 22       W_v col 40
    |                 |                 |
    v                 v                 |
+---v----+        +---v----+            |
|  RoPE  |        |  RoPE  |            |
+---+----+        +---+----+            |
    |                 |                 |
    |        +--------v----------+      |
    |        |     KV Cache      |<-----+   <-- V turns left into box
    |        |                   |
    |        +--------+----------+
    |                 |
```

### Quantize 단계 표시

```
   x_norm (FP16)
       |
  [INT4 quantize]  <-- per-token symmetric, clip 0.9
       |
   x_norm (INT4)
       |
       v
+------v------+
|  W_q (INT4) |
+-------------+
```

## 검증 체크리스트

다이어그램을 다 그렸으면 다음을 확인한다:

1. [ ] 박스 안에 한글/그리스 문자/wide 글리프가 없는가?
2. [ ] 모든 박스의 윗변 폭과 내부 라인 폭이 동일한가? (char count 일치)
3. [ ] 들어오는 `v`의 column과 나가는 `+`의 column이 같은가?
4. [ ] 다중 분기에서 split의 `+` column이 각 박스의 가운데 column과 일치하는가?
5. [ ] 평행 박스들이 같은 모양인가?
6. [ ] 모든 흐름선이 끊기지 않고 다음 노드/박스에 닿는가?
7. [ ] code fence가 ` ``` ` 으로 닫혀 있는가? (열림/닫힘 짝수)

검증을 자동화하려면:

```bash
# 다이어그램 라인 길이 확인 (정렬 디버깅)
awk 'NR>=START && NR<=END {printf "L%d|len=%d|%s\n", NR, length($0), $0}' file.md
```

특정 column에 문자가 일관되게 있는지 확인:

```python
# 모든 라인에서 col X의 문자 수집
[line[X] if len(line) > X else ' ' for line in block_lines]
```

## 안티패턴 모음 (이런 거 하지 마라)

| 안티패턴 | 왜 NG | 대안 |
|---|---|---|
| 박스 안에 한글 라벨 | wide char로 정렬 깨짐 | 영문 라벨 + 박스 밖 한글 주석 |
| Unicode `┌┐└┘─│` | 폰트별 렌더링 차이 | `+ - \|` |
| residual ring 외곽 박스 | 정렬 지옥 | `(+) <-- residual` 한 줄 |
| 분기 균등 간격으로 대충 | 박스 T-junction과 어긋남 | 박스 column에서 역산 |
| `+---v---+` 와 `+--+----+` 섞기 (v/+ 위치 다름) | 흐름선 zigzag | 같은 column으로 통일 |
| 박스 안에 긴 설명 욱여넣기 | 폭 일치 어려움 | 짧은 라벨 + 외부 설명 |
| 다이어그램 코드 블록에 언어 태그 | syntax highlight 충돌 | plain ``` |
| `--` `<--` `==` 들어간 본문에서 ligature 폰트 가정 | 폭이 달라질 수 있음 | code fence 안에서만 사용 |
| 박스 폭을 라벨이 빠듯하게 채움 | 비슷한 라벨로 1칸 overflow | 박스 폭에 1~2칸 여유 |
| `Q^T·diag(α)·W_q` 박스 안에 그대로 | `·` `α` 가 wide | `Q^T*diag(a)*W_q` ASCII로 |
| 범위 표기에 `~` (이중이든 단일이든) | 취소선으로 렌더될 수 있음 | `R1-R4` (하이픈) 또는 풀어쓰기 |

## 실전 작업 절차

1. **그릴 것을 정한다**: 박스 N개, 각 박스의 라벨, 박스 간 흐름의 토폴로지
2. **column 그리드를 먼저 정한다**: 박스 폭, gap, 박스 중심 column. 모든 다른 요소(split, 화살표, 박스 위치)는 이 그리드에서 역산
3. **박스 모양을 통일한다**: 같은 그룹의 박스는 동일한 폭/v 위치
4. **위에서 아래로 한 줄씩 그린다**: 각 줄에서 어느 column에 무엇이 있어야 할지 그리드 기준으로 결정
5. **박스 폭 검증**: 윗변/아랫변/내부 라인 폭이 같은지 char count로 확인
6. **흐름 검증**: 모든 `v`의 column = 아래 `+`/`v`의 column = 다음 `|`의 column 인지 확인
7. **markdown 뷰어로 확인**: 가능하면 실제 뷰어에서 시각 검토. ligature 폰트라면 비-ligature 폰트(예: Menlo)로도 확인

## 좋은 다이어그램의 한 줄 기준

> "코드 fence를 떼고 plain text로 봐도 모든 박스의 양 끝 `|`가 정확히 같은 column에 정렬되며, 박스 안에서 흐름선이 column을 점프하지 않는다."

이게 통과하면 거의 모든 markdown 뷰어에서 깨지지 않는다.
