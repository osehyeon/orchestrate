# 디버깅 템플릿

## 사용법

Claude Code에서 아래 프롬프트의 `{{변수}}`를 실제 값으로 바꿔서 입력한다.
가설이 떠오르지 않으면 변수 자리에 아무것도 적지 말고 **"원인 미상"** 이라고만 써라. 리드가 기본 3가설로 즉시 시작한다.

## 프롬프트

```
다음 버그를 조사하기 위한 에이전트 팀을 만들어줘.

**증상**: {{버그 증상 설명}}
**재현 조건**: {{재현 방법 — 알고 있다면}}
**의심되는 영역**: {{관련 파일/모듈 — 알고 있다면, 없으면 "없음"}}
**가설**: {{직접 가설을 쓰거나 "원인 미상"으로 남겨둔다}}

가설 배정 규칙 (리드가 적용):
- 사용자가 구체적 가설을 3개 제시한 경우 → 각 가설을 A/B/C에 그대로 배정
- 사용자가 1~2개만 제시한 경우 → 제시된 가설을 A(그리고 B)에 배정하고 나머지는 아래 기본 가설에서 채운다
- "원인 미상"인 경우 → 아래 3가지 기본 가설로 바로 시작

기본 가설 (원인 미상일 때):
- 가설A: "입력 데이터 문제" — 입력 검증, 데이터 형식, 경계값 관련 조사
- 가설B: "상태 관리 문제" — 경쟁 상태, 초기화 순서, 캐시/세션 상태 관련 조사
- 가설C: "외부 의존성 문제" — API 응답, 라이브러리 버전, 환경 변수 관련 조사

팀 구성 — 각자 다른 가설을 세우고 검증한다:
1. researcher 에이전트 타입으로 "가설A" 팀원을 스폰해줘 — 가설A 방향으로 조사한다.
2. researcher 에이전트 타입으로 "가설B" 팀원을 스폰해줘 — 가설B 방향으로 조사한다.
3. researcher 에이전트 타입으로 "가설C" 팀원을 스폰해줘 — 가설C 방향으로 조사한다.

리드(당신)는 각 조사자의 결과를 받으면 다른 조사자들에게 교차 공유해 반박·보완을 유도한다 (서브에이전트 간 직접 통신 불가).
리드는 조사 결과를 종합하여 가장 유력한 원인을 판단한다.
원인이 확정되면 implementer 에이전트 타입으로 "수정자"를 추가로 스폰해서 수정한다.
판단 기준: 2명 이상의 조사자가 동일 원인을 지목하거나, 재현 가능한 근거가 제시된 경우.
```

## 태스크 흐름

```
[가설A: 조사] ──┐
[가설B: 조사] ──┼──→ [리드가 결과 교차 배포] ──→ [각 조사자 반박 라운드] ──→ [리드: 원인 확정] ──→ [수정자: 버그 수정]
[가설C: 조사] ──┘

※ 병렬: 초기 가설 조사 3건은 독립적.
※ 직렬: 반박 라운드는 리드가 중계해야 시작 가능. 수정자는 원인 확정 후 투입.
※ 판단 기준을 충족하지 못하면 수정자를 스폰하지 말고 리드가 추가 가설을 제시한다.
```
