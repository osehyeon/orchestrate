# 설계/아키텍처 결정 템플릿

## 사용법

Claude Code에서 아래 프롬프트의 `{{변수}}`를 실제 값으로 바꿔서 입력한다.

구현이 포함되지 않고 **설계 결정·스키마·아키텍처 선택·리팩토링 방향**만 필요한 경우에 사용한다. 구현까지 필요하면 `feature.md`를 사용한다.

## 프롬프트

```
다음 설계를 결정하기 위한 에이전트 팀을 만들어줘.

**설계 주제**: {{설계할 대상 — 예: 주문 도메인 스키마, 결제 파이프라인 아키텍처}}
**결정이 필요한 이유**: {{배경 및 동기 — 성능 문제, 신규 요구사항, 기술 부채 등}}
**제약 조건**: {{호환성, 성능 목표, 마이그레이션 제한 등}}
**판단 기준**: {{단순성, 확장성, 운영비용, 팀 역량 등 우선순위}}

팀 구성:
1. architect 에이전트 타입으로 "설계자" 팀원을 스폰해줘 — 2~3개 후보 설계안을 도출하고, 각 안의 트레이드오프를 정리한다.
2. researcher 에이전트 타입으로 "조사자" 팀원을 스폰해줘 — 관련 선행 사례, 기존 코드베이스 제약, 유사 시스템의 선택을 조사한다.
3. devils-advocate 에이전트 타입으로 "반론자" 팀원을 스폰해줘 — 설계자가 제시한 각 후보의 숨은 리스크, 실패 모드, 대안 부재를 파고든다.

흐름:
- 설계자와 조사자가 병렬로 작업한다.
- 조사자의 결과가 도착하면 리드가 설계자에게 전달하여 후보를 보강하게 한다.
- 설계자 안이 확정되면 반론자가 검증한다.
- 리드는 반론자의 지적을 설계자에게 다시 전달해 최종안을 확정한다.
```

## 태스크 흐름

```
[설계자: 후보 설계안 N개]  ──┐
                              ├──(리드가 조사 결과 전달)──→ [설계자: 후보 보강] ──→ [반론자: 리스크 파헤치기] ──→ [리드: 최종 결정]
[조사자: 선행 사례 + 제약]  ──┘

※ 병렬: 설계자와 조사자는 독립.
※ 직렬: 반론자는 설계안 확정 후에만 의미가 있음.
※ 반론이 치명적이면 리드가 설계자에게 재검토를 요청한다 (루프 1회까지 허용).
```

## 결과물 구조 (권장)

경로는 CLAUDE.md의 "결과물 디렉토리 규칙"을 따른다 (`output/{작업슬러그}/` 하위).

```
output/{작업슬러그}/
├── decision.md            # 최종 결정 + 근거
├── candidates.md          # 고려한 후보 N개와 트레이드오프 표
├── research.md            # 선행 사례 및 제약 조사
└── risks.md               # 반론자가 지적한 리스크 및 완화 방안
```

## 변형

### 빠른 결정 (팀원 2명)
설계자 + 반론자. 조사 생략이 가능한 경우.

### 중대한 결정 (팀원 4~5명)
조사자를 2명으로 분리(선행사례/기존 코드베이스), 반론자 외에 reviewer를 추가해 구현 가능성 검증.
