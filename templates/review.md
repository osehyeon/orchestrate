# 코드 리뷰 템플릿

## 사용법

Claude Code에서 아래 프롬프트의 `{{변수}}`를 실제 값으로 바꿔서 입력한다.

## 프롬프트

```
다음 코드를 리뷰하기 위한 에이전트 팀을 만들어줘.

**리뷰 대상**: {{PR 번호 또는 파일/디렉토리 경로}}
**리뷰 목적**: {{일반 리뷰, 보안 감사, 성능 최적화 등}}
**핵심 관심사**: {{특별히 주의할 부분}}

팀 구성:
1. reviewer 에이전트 타입으로 "보안리뷰어" 팀원을 스폰해줘. 아래 체크리스트 전 항목을 점검한다:
   - 인증/인가 우회 가능성 (권한 상승, 세션 조작)
   - 입력 검증 누락 (SQL 인젝션, XSS, 커맨드 인젝션, 경로 순회)
   - 민감 정보 노출 (로그, 응답 본문, 에러 메시지에 시크릿 포함)
   - 의존성의 알려진 CVE 및 버전 핀닝
   - 암호화/해싱 오용 (약한 알고리즘, 예측 가능한 난수)

2. reviewer 에이전트 타입으로 "성능리뷰어" 팀원을 스폰해줘. 아래 체크리스트 전 항목을 점검한다:
   - N+1 쿼리, 불필요한 반복 호출
   - 과도한 메모리 할당 및 누수 가능성
   - 핫패스의 동기 I/O 및 블로킹 호출
   - 캐싱 기회 누락과 무효화 전략
   - 알고리즘 복잡도 (O(n²) 이상의 명시적 근거)

3. reviewer 에이전트 타입으로 "품질리뷰어" 팀원을 스폰해줘. 아래 체크리스트 전 항목을 점검한다:
   - 테스트 커버리지 (핵심 분기, 실패 경로, 경계 조건)
   - 네이밍/구조의 일관성 (기존 코드 스타일 준수)
   - 에러 처리의 일관성 (무음 실패, 중복 catch 방지)
   - 공개 API/타입 변경의 파급 범위
   - 문서·주석이 구현과 일치하는가

4. devils-advocate 에이전트 타입으로 "반론자" 팀원을 스폰해줘 — 세 리뷰어의 결론에 반론을 제기한다.
   - 각 리뷰어가 놓쳤을 가능성이 있는 관점
   - "괜찮다"고 판단한 항목의 숨은 리스크
   - 수정 제안이 다른 문제를 유발할 가능성
   - 리뷰 범위 자체가 좁게 설정된 것은 아닌지

세 리뷰어가 동시에 각자 체크리스트로 리뷰하고, 결과가 모이면 반론자가 검증한다.
리드는 반론자의 의견을 포함해 우선순위를 정리한다.
```

## 태스크 흐름

```
[보안리뷰어: 체크리스트 점검] ──┐
[성능리뷰어: 체크리스트 점검] ──┼──→ [반론자: 리뷰 결론 반박] ──→ [리드: 우선순위 종합]
[품질리뷰어: 체크리스트 점검] ──┘

※ 세 리뷰어는 독립적 병렬 실행 가능 (대상 파일이 동일해도 읽기 전용이라 충돌 없음).
※ 반론자는 세 리뷰어의 출력이 모두 도착한 후 시작 (직렬 의존).
```
