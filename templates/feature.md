# 코드 기능 개발 템플릿

## 사용법

Claude Code에서 아래 프롬프트의 `{{변수}}`를 실제 값으로 바꿔서 입력한다.

## 프롬프트

```
다음 기능을 개발하기 위한 에이전트 팀을 만들어줘.

**기능 목표**: {{기능 설명}}
**대상 프로젝트**: {{프로젝트 경로 또는 설명}}
**기술 스택**: {{사용하는 언어/프레임워크}}
**제약 조건**: {{일정, 성능 요구사항 등}}

팀 구성:
1. architect 에이전트 타입으로 "설계자" 팀원을 스폰해줘 — 기능의 아키텍처와 모듈 경계를 설계하고, 구현 계획과 공개 인터페이스를 확정한다. plan approval을 요구해줘 — 리드가 타당성을 판단하여 승인/거부한다.
2. researcher 에이전트 타입으로 "조사자" 팀원을 스폰해줘 — 기존 코드베이스를 분석하고, 영향 받는 파일과 패턴을 파악한다.

설계자의 계획이 승인되면 리드는 구현자 각각에게 owned_paths를 명시해 스폰한다:

3. implementer 에이전트 타입으로 "구현자A" 팀원을 스폰해줘.
   - owned_paths: {{구현자A가 수정 가능한 파일/디렉토리 목록 — 예: src/feature/core.ts, src/feature/types.ts}}
   - 담당: 핵심 로직 구현. 설계자가 확정한 인터페이스에 맞춰 작성한다.

4. implementer 에이전트 타입으로 "구현자B" 팀원을 스폰해줘.
   - owned_paths: {{구현자B가 수정 가능한 파일/디렉토리 목록 — 예: src/feature/*.test.ts, tests/feature/}}
   - 담당: 테스트 코드 작성. 설계자의 인터페이스 계약만 의존하므로 구현자A와 병렬 가능.

5. reviewer 에이전트 타입으로 "검증자" 팀원을 스폰해줘 — 구현이 완료되면 코드 리뷰를 수행한다.

**파일 충돌 방지 규칙 (리드가 강제)**:
- 각 구현자는 자신의 owned_paths 외 경로를 Edit/Write 할 수 없다.
- owned_paths가 겹치면 리드가 스폰 전에 분리한다.
- 공통 유틸 등 공유 영역이 필요하면 설계자가 설계 단계에서 별도 파일로 분리하고, 구현자 중 한 명에게만 배정한다.
- 구현자가 경계 밖 변경이 필요하다고 판단하면 리드에게 메시지로 요청한다.
```

## 태스크 흐름

```
[설계자: 아키텍처 + 인터페이스 확정] ──┐
                                       ├──(인터페이스 승인)──→ [구현자A: 핵심 로직] ──┐
[조사자: 코드베이스 분석]             ──┘                      [구현자B: 테스트]    ──┤──→ [검증자: 코드 리뷰]
                                                                                      │
                                                                                      └──→ [리드: 통합 및 정리]

※ 설계자·조사자는 독립 병렬.
※ 구현자A·B는 설계자의 인터페이스 확정 후에만 병렬 가능. 인터페이스가 바뀌면 양쪽 모두 재작업.
※ owned_paths가 분리된 경우에만 진짜 병렬. 겹치면 직렬로 떨어뜨려라.
```

## 변형

### 소규모 기능 (팀원 3명)
설계자 + 구현자 + 검증자로 축소 가능

### 대규모 기능 (팀원 6-7명)
구현자를 프론트엔드/백엔드/DB로 분리 가능. 각 구현자의 owned_paths를 레이어별로 명확히 구분할 것.
